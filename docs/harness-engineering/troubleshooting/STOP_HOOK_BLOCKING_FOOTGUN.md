# Stop 훅을 blocking으로 두면 모든 턴이 막힌다 (무한루프 포함)

## 문제

`Stop` 훅으로 품질 게이트(`check-stubs.sh`)를 걸 때, 처음엔 마커 발견 시 `{"decision":"block","reason":...}`를 반환해 에이전트를 강제로 다시 일하게 만들려 했다. 그런데 두 가지가 터진다.

1. 백엔드 트리에 실제로 `// 임시` 마커가 1건 있어(작동하는 코드의 설명 코멘트), blocking으로 두면 **사용자가 단순 질문만 해도 매 턴이 막힌다.**
2. block이 에이전트를 다시 깨우고, 그 에이전트가 또 Stop하면 Stop 훅이 또 block → **무한 재호출** 위험.

## 원인

- `Stop` 훅은 순수 대화 턴 포함 **모든 종료 시점**에 발화한다. 트리 전체를 스캔하는 게이트를 무조건 block으로 두면, 이번 턴이 코드와 무관해도 "트리에 마커가 있다"는 이유로 막힌다.
- block은 모델을 재개시키는데, 재개된 모델이 또 Stop하면 같은 훅이 또 평가된다. 종료 조건이 없으면 루프.

## 해결

1. **기본값을 warn(non-blocking)으로.** systemMessage로 사용자에게 알리되 턴은 막지 않는다.
   ```bash
   jq -cn --arg msg "$MSG" '{systemMessage:$msg}'   # block 아님
   ```
2. **재진입 가드.** Stop 훅 입력의 `stop_hook_active`가 true면(이미 Stop 훅 때문에 재개 중) 즉시 통과해 루프를 끊는다.
   ```bash
   [[ "$(jq -r '.stop_hook_active // false')" == "true" ]] && exit 0
   ```
3. **하드 게이트는 분리.** `check-stubs.sh`(인자 없이)는 발견 시 `exit 1`인 CI/pre-commit용 하드 게이트. 인터랙티브 경고와 CI 차단을 **다른 모드**로 분리해, 인터랙티브에선 안 가두고 CI에선 확실히 막는다.
4. 트리가 마커-free가 된 뒤에는 스크립트 주석 한 줄로 blocking 래칫으로 승격 가능(회귀 방지).

## CS 원리

**재진입성(reentrancy)과 멱등성.** 자기 출력이 자기를 다시 트리거할 수 있는 훅은, 진행 상태 플래그(`stop_hook_active`)로 재진입을 끊지 않으면 발산한다 — 인터럽트 핸들러가 자기 인터럽트를 다시 거는 것과 같은 함정. 또한 **부수효과(턴 차단)는 트리거 범위와 일치해야 한다**: "이번 변경"이 아니라 "트리 전체 상태"로 차단하면, 무관한 컨텍스트까지 막는 과차단(over-blocking)이 된다. 그래서 인터랙티브 경고(soft)와 CI 게이트(hard)를 분리하는 게 정석이다.
