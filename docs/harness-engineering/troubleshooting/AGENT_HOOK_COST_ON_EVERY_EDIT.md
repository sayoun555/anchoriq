# PostToolUse 의미 리뷰어가 매 편집마다 비싸다

## 문제

의미 적대자(PostToolUse `type:"agent"` 훅)를 `matcher:"Edit|Write"`로 걸면, **모든 파일 편집마다 에이전트가 한 번씩 뜬다.** 코딩 세션은 편집이 수십~수백 번이라:
- 매 편집에 LLM 호출 → 루프가 느려짐(편집당 수 초)
- 문서·설정·프론트 편집에도 발화 → 불필요한 비용

품질 게이트가 좋아도 **개발 루프를 끈적하게** 만들면 안 쓰게 된다.

## 원인

- `PostToolUse`는 매칭되는 도구 호출마다 발화한다. agent 훅은 그때마다 에이전트(기본 Haiku)를 스폰한다.
- 백엔드 `.java`만 의미 리뷰가 필요한데, matcher는 모든 `Edit|Write`를 잡는다 → 대상 외 편집까지 에이전트가 떠서 빠른 종료라도 오버헤드가 쌓인다.

## 해결

여러 노브를 조합한다(현재 적용 + 대안):

1. **프롬프트 0단계 빠른 종료(적용)**: `review-protocol.md` 첫 단계에서 대상이 `backend/anchoriq-*/src/main/java/**/*.java`가 아니면 *아무것도 읽지 않고* 즉시 통과. 대상 외 편집의 비용을 최소화.
2. **저비용 모델 기본(적용)**: 기본 Haiku로 고빈도·저비용 운용. DDD 적발률이 더 필요하면 훅에 `"model":"claude-sonnet-4-6"` 추가(비용↑).
3. **빈도 축소(대안)**: matcher를 `Write`(신규 파일 생성)로 좁힘 — 아키텍처 위반(잘못된 레이어에 새 클래스, 인터페이스 누락)은 대부분 *파일 생성* 시점에 들어오므로, 가장 값진 순간만 잡고 편집당 비용을 없앤다.
4. **배치 리뷰로 이동(대안)**: 편집마다(N회)가 아니라 **Stop 시점에 1회**, 이번 턴에 바뀐 파일을 묶어 리뷰. wall-clock·비용을 N→1로. 전파는 동일하게 채널 A(Stop reason).

## CS 원리

**검증의 입도(granularity)와 비용-지연 트레이드오프.** "매 변경마다 검증"(eager)은 피드백이 빠르지만 단가가 높고, "경계에서 한 번 검증"(batched/lazy)은 싸지만 피드백이 늦다 — CI에서 *pre-commit 훅 vs PR 게이트*의 선택과 같다. 정답은 **신호 가치가 가장 높은 지점에 검증을 배치**하는 것: 아키텍처 위반은 파일 생성 시점에 집중되므로 `Write`에 좁히는 게 비용 대비 효과가 크다. 또한 비싼 검증 앞에 **싼 사전 필터**(확장자 게이트, 빠른 종료)를 두는 건 정석적인 단계적 비용 절감(cascading filters)이다.
