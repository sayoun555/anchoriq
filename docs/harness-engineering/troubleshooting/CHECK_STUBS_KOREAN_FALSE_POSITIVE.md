# anti-stub 게이트의 한국어 코멘트 오탐

## 문제

`check-stubs.sh`를 처음 만들 때 거친 패턴(`// *placeholder`, `// *simulate` 등)으로 백엔드를 스캔하니 **0건**이 나왔다. "트리가 깨끗하다"고 판단하고 Stop 게이트를 blocking으로 두려 했다. 그런데 정식 패턴(`임시|미구현|추후|일단` 포함)으로 다시 돌리니 **1건**이 잡혔다:

```
NotificationDispatcherImpl.java:79:  // 임시로 conditionValue를 destination으로 사용하지 않고, 기본 destination 사용
```

즉 (1) 첫 스캔이 마커를 놓쳤고, (2) 잡힌 그 마커는 *크래시 stub*이 아니라 **작동하는 코드의 설명 코멘트**였다.

## 원인

- **놓친 이유**: 첫 패턴에 한국어 마커(`임시`)가 없었다. 영어 마커만 보면 한국어 주석으로 표현된 기술부채를 통째로 놓친다.
- **오탐 위험**: `임시`/`일단`/`추후`는 "미완성 stub"일 수도, "현재는 이렇게 동작한다"는 정상 설명일 수도 있다. 위 사례는 destination 해석을 의도적으로 단순화한 **동작하는** 코드였다 — AGENTS.md 규칙 10이 경계하는 임시방편은 맞지만, 빌드를 깨는 stub은 아니다.

## 해결

1. **마커 집합에 한국어 포함**(`임시|미구현|추후|일단`) — 한국어 코드베이스의 기술부채를 놓치지 않게.
2. **코멘트 앵커 기반 정규식**으로 식별자 오탐 차단:
   ```
   (//|/\*|\*)[[:space:]]*(TODO|FIXME|...|임시|미구현|...)
   ```
   `임시저장(tempSave)` 같은 식별자가 아니라, 주석에 쓰인 마커만 잡는다.
3. **soft 마커는 warn으로 처리**. `임시` 같은 약한 신호는 매 턴 차단하지 않고 경고만. 강한 신호(`throw new UnsupportedOperationException`, `Not implemented`)와 약한 신호를 같은 게이트에 두되, 게이트 자체를 기본 non-blocking으로(→ [STOP_HOOK_BLOCKING_FOOTGUN.md](STOP_HOOK_BLOCKING_FOOTGUN.md)).
4. 실제로 잡힌 1건은 working code라 즉시 수정하지 않고 사용자에게 보고 — "정석 수정 시 Stop을 blocking으로 승격 가능"으로 남김.

## CS 원리

**휴리스틱 탐지의 정밀도-재현율(precision-recall) 트레이드오프.** 마커를 늘리면 재현율↑(놓침↓)이지만 정밀도↓(오탐↑). 키워드 게이트는 본질적으로 근사다. 그래서 (a) 앵커링으로 정밀도를 보강하고, (b) 신호 강도에 따라 **차단 vs 경고**로 대응을 차등화하고, (c) 최종 판정은 사람/추론 적대자에게 넘긴다. 정적 분석에서 "린트 경고 ≠ 컴파일 에러"로 심각도를 나누는 것과 같은 설계다. 그리고 다국어 코드베이스에서 **탐지 사전은 코드의 자연어를 따라가야 한다** — 영어만 가정하면 체계적으로 놓친다.
