# `paste -d ' · '` 가 구분자를 뒤섞는다

## 문제

`governing-doc.sh`에서 지배 문서 목록을 ` · `로 이어 한 줄로 만들려고 `paste -sd ' · '`를 썼더니, 3개 항목 출력이 이상하게 섞였다:

```
plan/ARCHITECTURE.md(...) plan/DB_OPTIMIZATION.md(Neo4j)·plan/AGENTS.md(...)
                         ^^                              ^^
                         첫 구분자는 공백, 둘째는 '·'  ← 불일치
```

원하던 건 `A · B · C`인데 `A B·C`가 나왔다.

## 원인

`paste -d`의 인자는 **구분자 한 개**가 아니라 **구분자 문자들의 리스트**다. 항목 사이마다 그 리스트의 문자를 **순환**해서 쓴다. `' · '`(공백·`·`·공백, 3문자)를 주면:
- 1번째 구분자 = `' '`(공백)
- 2번째 구분자 = `'·'`
- 3번째 구분자 = `' '`(공백) …

즉 ` · `를 "하나의 멀티문자 구분자"로 해석하지 않는다. POSIX `paste`는 멀티문자 구분자를 지원하지 않는다.

## 해결

멀티문자 구분자가 필요하면 `paste` 대신 `awk`로 직접 결합한다(동시에 중복 제거):

```bash
JOINED="$(printf '%s\n' "${docs[@]}" \
  | awk '!seen[$0]++{a[++n]=$0} END{for(i=1;i<=n;i++) printf "%s%s",(i>1?" · ":""),a[i]}')"
```
- `!seen[$0]++` 로 dedup
- 첫 항목 앞엔 구분자 없이, 이후 항목 앞에 `" · "` 부착

결과: `A · B · C` (일관).

## CS 원리

**POSIX 툴의 인자 의미론을 가정하지 말고 확인하라.** `paste -d`, `tr`, `cut -d` 등 여러 텍스트 툴이 "구분자 인자"를 **문자 집합/리스트**로 해석한다(단일 문자열이 아니라). 직관과 어긋나는 대표 사례:
- `cut -d` : 단일 **문자**만 허용(멀티문자 구분자 불가)
- `tr` : 문자 **집합** 대응
- `paste -d` : 문자 **리스트**를 순환

문자열 조인처럼 "당연해 보이는" 연산도 셸 툴에선 의외의 모델을 가진다. 파이프 테스트로 **출력 형태를 눈으로 확인**하는 절차가 이런 조용한 버그를 잡는다.
