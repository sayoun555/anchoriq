# DRIFT — "에이전트가 설계문서를 무시할 때 하네스가 강세 아닌가?" (사용자 가설 직접 검증)

> 동기: 사용자 지적 — "구현하다 보면 에이전트가 설계문서를 *무시*하는 경우가 있고, 하네스는 *그때* 강세를 보이는 것 아니냐." 옳은 지적. 이전 실험(BOUNDARY 등)은 둘 다 *성실한* 에이전트라 하네스의 강제 기능이 발휘될 *무시 조건*을 못 만들었다.
> 설계: 에이전트의 *자신만만한 디폴트가 틀린* 규칙을 고른다 → CLAUDE.md 8번 "외부 API는 @Transactional 안에서 호출 금지 → 보상 트랜잭션". 디폴트(일관성 위해 통째 @Transactional)는 *틀림*. "데이터 일관성 보장"이라는 **함정 프레이밍**으로 드리프트 유도.
> 조작: Arm A = 편집 시점 거버닝-닥 리마인더(TRANSACTION_DESIGN.md 짚어줌, =PreToolUse 훅). Arm B = 리마인더 없음. 둘 다 ambient(CLAUDE.md 8번을 *산문으로* 보유) → **"산문은 씹히는데 훅은 강제한다" 명제 테스트.** 약한 모델에서 드리프트 가능성↑ → Opus + Haiku 양 티어.

## 발견

1. **핵심 비추론 규칙(외부 API @Transactional 밖)은 4개 전부 준수 — 드리프트 안 일어남.** 약한 모델(Haiku)조차, 리마인더 없이도, 함정 프레이밍에도 안 빠졌다. Haiku Arm B 출력: *"based on my reading of the existing codebase…"* — 스스로 기존 코드를 읽고 따랐다. **유능/반쯤유능 모델은 집중 과제에서 *기본적으로 조사*한다 → "무시" 조건이 잘 안 생긴다.**
2. **품질 차이는 훅이 아니라 *모델 티어*에서 났다.** Opus는 self-invocation 함정을 둘 다 회피("프록시 우회되니 트랜잭션 메서드는 별도 빈에"), Haiku는 둘 다 걸렸다. 그리고 **훅(Arm A)이 Haiku의 그 버그를 못 막았다** — 리마인더 받은 Haiku Arm A도 동일 버그. 리마인더는 *인식*은 줘도(주석은 정확) *역량 격차*는 못 메웠다.
3. **사용자 가설은 이론적으론 옳지만, 내가 만들 수 있는 조건(집중 과제, Opus·Haiku)에선 드리프트가 안 났다.** 진짜 "무시" 상태(긴 세션·산만·속도압박·더 약한 모델)는 신선한 집중 서브에이전트로 재현이 안 된다 → 그 조건에서의 하네스 값은 *여전히 미측정(반증도 안 됨)*.

## Stage 2 — 그래서 *다른 절반*을 쟀다: PostToolUse 적대자가 그 버그를 잡나? ★ POSITIVE

Stage 1은 PreToolUse *주입*만 쟀다(=0). 하네스의 다른 절반 — **PostToolUse 의미 적대자**(코드 작성 *후* 위반 검사) — 에 Haiku의 버그 코드를 **블라인드로**(무슨 버그인지 안 알려주고) 던졌다.

| 적대자 실행 | self-invocation @Transactional 버그 | 기타 |
|---|---|---|
| 1차 (프롬프트에 "트랜잭션 실작동 따져라" 힌트有) | **[HIGH] 적발 — block** | 환불실패-C 누락·멱등성·게이트웨이 라우팅 누수 3건 |
| **2차 (힌트 제거, 순수 블라인드)** | **[HIGH] #1 차단 사유로 독립 적발** | 환불-C·게이트웨이 중복 2건 |

2차(힌트 없음)에서도 적대자는 self-invocation 버그를 *최우선 차단 사유*로 잡았고, **기존 코드의 정답 패턴(`PaymentTransactionService` public @Transactional 별도 빈)을 스스로 찾아 "이걸로 고쳐라"** 까지 제시했다. 심지어 *설계문서 예시 자체가 `@Transactional private`로 잘못 적혀 있음*을 지적("그건 의사코드일 뿐, 실제 표준은 PaymentTransactionService")했다.

→ **하네스가 *못 막은* 버그(PreToolUse·리마인더)를, 하네스가 *잡았다*(PostToolUse·적대자).** 검증-측에서 처음으로 명확한 양수. 약한 모델이 실제로 낸, 데이터 정합성을 깨는 미묘한 버그를 독립적으로 적발.

## ⚠️ 한계 (정직)

- **사전등록 없는 탐색적 실험**(사용자 대화 중 즉흥). n=1 버그·1 코드샘플. "적대자가 *이* 버그를 잡았다"지 "모든 버그를 낮은 오탐으로 잡는다"의 증명 아님. (단 기존 DETECT/H4b: 적대자는 진짜 DDD 위반을 잡고 과탐 안 함 — 일관됨.)
- **그 버그엔 "tell"이 있었다**: repo에 *정답 패턴*(PaymentTransactionService)이 존재해 적대자가 대조했다. 정답 예시가 없는 repo에선 덜 확실.
- **적대자는 하네스에서 제일 비싼 부품**(편집마다 에이전트 1).
- "무시" 조건 미재현(긴/산만/속도압박): 사용자 가설의 *생성-측* 핵심은 여전히 열림. 단 *검증-측*에선 답을 얻음.

## 한 줄

> 사용자 가설("에이전트가 문서 무시할 때 하네스 강세")을 정조준 — 생성-측에선 **드리프트 0**(Opus·Haiku 모두 핵심 규칙 준수, 리마인더 무관, 품질차는 모델티어). **그러나 결정적으로**: 리마인더가 *못 막은* Haiku의 self-invocation 트랜잭션 버그를, **하네스의 PostToolUse 적대자가 블라인드로 #1 차단 사유로 잡았다(힌트 제거 후에도).** → **하네스의 값은 "사전에 알려주기"(주입 ≈0)가 아니라 "사후에 잡기"(적대 리뷰 = 양수)에 있다.** 사용자의 직관은 옳았다 — 단 메커니즘이 *리마인더*가 아니라 *적대적 검증*이었다. 이는 프로젝트 연구근거와 정합(self-critique 약함·외부 검증/Agent-as-Judge 강함, v-next 런타임 grounding 양수)하며, 하네스의 무게중심을 **강제·검증 절반(PostToolUse 적대자 + 결정론 게이트 + 런타임 grounding)** 으로 옮겨야 함을 시사. 정보-전달 절반(SessionStart/PreToolUse 주입)은 유능 모델에 redundant.
