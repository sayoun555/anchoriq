# v3 (taster) — runtime 계산 버그: 실행 vs 암산

> 가설: 컴파일은 멀쩡하지만 *출력이 비자명한* 계산은, 읽기(암산)로는 불안정하고 실행(jshell)으로만 확정된다 → grounding이 reading을 이긴다(H1~H4의 천장을 깰 "역량 경계").
> 출처 맥락: Huang ICLR'24 + H3 결론(grounding 이득은 역량 경계에서만). "고도화 테스트 나중에" 전 **맛보기 한 개**.

## 결론 (정직하게 — 또 천장)

1. **v3 taster도 미지지.** 의도적으로 깐 20스텝 noise를 intrinsic이 **순진한 추적이 아니라 영리한 분해로** 정확히 풀었다(49). grounded와 동일.
2. **역량 경계가 예상보다 멀다**: *결정론적·자기완결* 계산은 — 아무리 스텝이 많아도 — 모델이 **신뢰성 있는 지름길(그룹핑)을 찾아** 추론으로 푼다. 단순 길이로는 경계에 못 닿는다.
3. **grounded는 설계대로 작동**: 실행 렌즈가 jshell 실행 + Python 교차검증까지 = audit-grade 증거. **정확도는 같지만 재현 가능한 증거**를 남김(H1~H4와 동일한 패턴).

## 진짜 역량 경계는 어디인가 (v-next 설계)

결정론적 코드로는 못 닿는다. grounding이 reading을 *정확도로* 이기려면:
- **비결정성**: 동시성/경합, 타이밍, 난수 — 읽기로 원리상 예측 불가.
- **환경/IO 의존**: 실제 DB 상태, 네트워크, 파일시스템, *라이브러리 버전별* 실제 동작.
- **진짜 난해한 계산**: 거대 입력, 복잡 수치(부동소수점 누적 오차 등).

→ 이건 **단위 fixture로 못 만든다**(결정론·자기완결이라). 진짜 v-next는 *실행 환경*이 필요 — "고도화 테스트 나중에 한번에"의 핵심 과제.

## 한 줄

> 20스텝 계산도 모델은 분해로 정확히 풀었다 — **결정론적 readable 코드의 역량 경계는 매우 멀다.** grounding의 *정확도* 이득은 **비결정성·환경의존·진짜 난해함**에서만 날 것이고, 그건 단위 fixture가 아니라 실행 환경을 요구한다. (taster 결론: 천장은 우리 생각보다 높다.)
