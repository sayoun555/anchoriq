# H3 — grounding 제거 시 악화되나 (grounded vs intrinsic)

> 가설: 외부 실행 신호(컴파일)를 빼고 추론만 하면(intrinsic) 검증이 악화된다.
> 출처: Huang ICLR'24(외부 피드백 없으면 self-correct 악화) · Kamoi TACL'24. 카탈로그: [../HYPOTHESES.md](../HYPOTHESES.md#h3)
> 설계: build-level finding(컴파일 신호가 결정적)으로. **의도적 컴파일 에러를 심음** — `_EvalTypeBug.demo()`가 존재하지 않는 `getDestination()` 호출(실제는 `destinationFor`). *읽으면 그럴듯하지만* 컴파일러는 잡음. api(leaf 모듈)에 둬 core는 안 깨지게.

## 결론 (정직하게 — 예측 빗나감)

1. **H3 미지지.** intrinsic(컴파일 안 돌림)이 grounded와 **동일하게 2/2.** 진짜 컴파일 에러를 *놓치지 않았다.* 예측(intrinsic이 놓침)이 틀렸다.
2. **왜**: 에러가 **모델 prior 안**이었다. intrinsic 에이전트가 NotificationSettings를 읽고 *"@Getter는 파라미터 있는 getDestination을 생성하지 않는다 → cannot find symbol"* 이라고 **컴파일 없이 정확히 추론**. 즉 Lombok 의미론을 모델이 이미 알아서 실행 신호가 불필요했다.
3. **grounding이 산 것 = 감사 가능한 증거 + (역설적) 비용**: grounded 렌즈들은 실제 `javac`/`gradlew --rerun-tasks`를 돌려 *정확한 에러 메시지를 재현*(audit-grade). 정확도는 같지만 **토큰 3배·시간 2.4배.** 이 케이스에선 grounding이 비싼 중복.

## 3실험(H1·H2·H3) 관통 패턴

| | 측정된 정확도 차 | 측정된 grounding 이득 |
|---|---|---|
| H1 | 없음 | 추적가능성 |
| H2 | 없음 | 근거 다양성 |
| H3 | 없음 | 감사 증거(비용 3x) |

> **반복 확인된 것**: 우리 테스트 에러들이 전부 **모델 역량 *안***이라, grounding의 측정 이득은 정확도가 아니라 **감사가능성**이고 종종 **비용 초과**다. **grounding의 정확도 이득은 "모델 역량 경계"에서만 나타날 텐데, 우리 데이터셋이 아직 거기 못 닿았다.** ← 이게 3실험의 진짜 결론.

## 다음 실험 (v3) — 역량 경계로

intrinsic 추론이 *틀릴* 에러가 필요:
- **prior 밖 컴파일 에러**: 복잡한 제네릭 추론, 오버로드 모호성, 버전별 API 동작.
- **행위(runtime) 버그**: 읽기로는 *원리상* 검증 불가(테스트 실행 필요). 예: off-by-one, 동시성, 잘못된 SQL. 여기서 grounding(테스트 실행)이 reading을 확실히 이길 것.

## 한 줄

> intrinsic이 컴파일 없이도 잡았다 — 에러가 모델 prior 안이라. **grounding의 정확도 이득은 "모델이 추론으로 못 푸는" 경계에서만 측정될 것.** 3번 연속 "미지지"는 실패가 아니라, *우리 데이터가 너무 쉬웠다*는 일관된 측정 — v3는 역량 경계(prior 밖 컴파일/runtime 버그)로 가야 한다.
