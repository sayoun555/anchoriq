# H8 — 테스트/실행 vs 정적 추론 (라이브러리 동작 edge)

> 가설: 비자명한 *라이브러리 동작*은 읽기/암기로 틀리기 쉽고 실행(테스트)으로만 확정 → 실행 기반 검증이 정적 추론을 이긴다.
> 출처: Chen ICLR'24 Self-Debug(실행 피드백). 카탈로그: [../HYPOTHESES.md](../HYPOTHESES.md#h8)
> 설계: v3(산술)는 모델이 분해로 풀어 천장 → 이번엔 **Java `String.split` trailing-empty 제거**(많은 사람이 틀리는 edge). finding(bogus): `splitCount("a,b,,")==4` (실제=2).

## ⚠️ 측정 오염 발견·수정 (방법론 교훈)

**1차 intrinsic run은 오염됐다**: 내가 fixture *주석*에 "trailing empty 제거", "많은 사람이 4라 답함"이라 적어 **정답을 leak**. intrinsic 에이전트가 파일을 Read하며 그 주석을 근거로 댔다. → **주석을 중립으로 정리(정답 제거)하고 재실행** → 그래도 refute(=주석 없이도 모델이 진짜 알고 있었음). 오염을 잡고 보정함.

> **eval 교훈: fixture/주석이 검증자에게 ground truth를 leak할 수 있다.** 검증 대상에 정답 단서를 두지 말 것. (troubleshooting급 방법론 함정.)

## 결론 — 10실험 capstone 강화

산술(v3)에 이어 **라이브러리 동작 edge(H8)까지 모델이 안다** → 정적 추론으로 충분, 실행/테스트가 정확도를 못 더한다. **"모델 역량 경계"가 라이브러리 의미론까지 포함할 만큼 멀다.**

10실험 전부(H1·H2·H3·H4·H4b·v3·DETECT·H11·H8) **verdict 정확도 차 0.** 신호가 난 곳은 *confidence*(H4b)와 *루브릭의 비존재 규칙 적발*(H11)뿐. **grounding/구조의 값 = 일관되게 auditability/confidence이지 정확도가 아니다.**

## 진짜 경계 (재확인)

정확도 신호는 **모델이 추론으로 못 푸는 것** = 비결정성(동시성/타이밍/난수)·환경의존(실제 DB/네트워크 상태)에서만. 결정론 코드·표준 라이브러리는 다 추론 가능 → **실행 환경이 필요한 v-next**가 유일한 미답.

## 한 줄

> 라이브러리 edge(Java split)도 모델이 알아 — **10실험 내내 정확도는 안 변했다.** grounding은 *감사근거*(JDK 스펙 인용)를 더했을 뿐. 그리고 이번엔 **내 fixture가 정답을 leak한 오염을 잡아 보정**했다(eval 자체의 신뢰성 관리).
