# H1 — 도구 grounding vs 순수 읽기

> 가설: 검증자에게 도구(컴파일·grep·검색)를 주면 *그럴듯하지만 거짓인* 위반 주장을 순수 읽기보다 덜 hallucinate한다(false-confirm↓).
> 출처: CRITIC ICLR'24 · Agent-as-a-Judge Meta'24. 카탈로그: [../HYPOTHESES.md](../HYPOTHESES.md#h1)

## 결론 (정직하게)

1. **H1, 이 난이도에선 미지지(not supported).** 두 패널 다 bogus 4건을 전부 정확히 refute, false-confirm 0. **정확도 차이 없음.**
2. **실험 confound 발견**: "read" 패널이 *순수 읽기가 아니었다* — 기본 툴셋에 grep/Bash가 있어 read 에이전트도 `grep getConditionValue=0`을 근거로 댔다. **깨끗한 A/B가 아니었다.**
3. **grounding이 *측정상* 산 것 = 추적가능성(auditability)**: grounded는 12/12 판정이 명시적 도구 출력(ground-check "✅ compileJava 통과", grep exit code)을 인용. 정확도가 같아도 **감사·신뢰 측면 이득**.
4. **비용**: grounded가 +4.5% 토큰, +61% 도구호출, +24% 시간. 쉬운 케이스에선 정확도 이득 없이 오버헤드.

## 다음 실험 (v2 설계)

- **(a) 더 어려운 bogus** — 순수 읽기가 *실제로 hallucinate*할 케이스(길고 spec과 미묘하게 어긋나는). 현 데이터셋은 너무 쉬워 신호가 천장에 막힘.
- **(b) read 패널 도구 제한** — 깨끗한 A/B를 위해 read 변종은 grep/Bash deny(Read만). 그래야 "도구 있음 vs 없음"이 진짜 변수.
- **(c) real 위반 추가** — detection율 측정(현재 specificity만). 모듈 내 fixture seed.
- **(d) n 키우기** — 4 → 12+.

## 한 줄

> **n=1 일화 → 첫 측정값.** "grounded가 낫다"는 *이 난이도/이 설계에선 정확도로 입증 안 됨*; 이득은 추적가능성, 비용은 +24% 시간. 신호를 보려면 v2(더 어렵게 + 도구 제한)가 필요하다.
