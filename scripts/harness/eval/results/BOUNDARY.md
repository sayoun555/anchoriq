# BOUNDARY — 비추론 계약에서 하네스의 한계효용 (그리고 "정적 정보" 경계의 확정)

> 사전등록: [`../dataset-boundary.json`](../dataset-boundary.json) (생성 전 작성).
> 목적: COMPLIANCE는 *추론 가능* 규칙을 골라 천장(null)을 쳤다. 이번엔 *의도적으로 추론 불가*한 임의 계약(Kafka 파티션 수·컨슈머그룹명·DLT 접미사·키 필드명)을 골라, 하네스의 거버닝-닥 주입이 앰비언트보다 준수를 올리는 *경계*가 실재하는지 측정.

## Run 1 — 2026-06-09 · 1 task × 2 arm, 블라인드(코드 출력만, 파일 미작성)

대상 계약(KAFKA_DESIGN.md): ais-positions = 파티션 **3** · 키 **mmsi** · 그룹 **ais-redis-writer** · DLT **`.DLT`** · 재시도 **FixedBackOff(1000L, 3)**.

## 왜 — confound이 곧 발견

두 arm의 출력 코드가 둘 다 `// KAFKA_DESIGN.md ...` 주석을 달고, 둘 다 "기존 구현이 이미 있다"고 보고했다. grep으로 확인한 결과 — **내가 "비추론"이라 고른 임의값이 *기존 코드에 전부 박혀 있었다*:**
- `config/KafkaTopicConfig.java:34` → `.partitions(3)`
- `producer/AisKafkaProducer.java:26` → `send(String mmsi, …)`
- `consumer/ais/AisRedisConsumer.java:38` → `groupId = "ais-redis-writer"`
- `config/KafkaConsumerConfig.java:54` → `new FixedBackOff(1000L, 3)` + `DeadLetterPublishingRecoverer`

→ 그 값들은 *일반 지식*으론 비추론이 맞지만, **repo에 정적으로 존재**했다. Arm B(주입 없음)도 CLAUDE.md의 "설계문서 먼저 읽기" 의무 + repo 접근으로 **문서와 기존 코드를 스스로 회수**해 그대로 따랐다. 하네스가 짚어주는 것이 *불필요(redundant)*했다.

## 결론 — "정적 정보" 경계의 확정 (4실험 수렴)

| 실험 | 정보의 소재 | 하네스 한계효용 |
|---|---|---|
| capstone(11) | 모델 일반 지식으로 추론 가능 | **0** |
| COMPLIANCE | repo의 CLAUDE.md + DDD 컨벤션(정적) | **0** |
| **BOUNDARY(이번)** | **설계문서 + 기존 코드(정적, 회수 가능)** | **0** |
| **v-next** | **런타임 DB 행(어떤 정적 소스에도 없음)** | **+50%p (유일 정답 경로)** |

**법칙(증거기반):** 하네스(=정적 설계정보를 컨텍스트에 주입하는 메커니즘)의 측정된 한계효용은
> **필요한 정보가 모델이 읽을 수 있는 *정적 소스 어디에도 없을 때*에만 양수.**
정적으로 존재(코드·문서·일반지식)하면 유능한 모델이 *알아서 회수* → 주입은 redundant. 정보가 정적 소스에 없을 때(런타임 상태·환경·아직 안 짠 신규 결정)에만 "컨텍스트에 넣어주는" 행위가 정답을 가른다 — 그게 v-next에서 grounding이, 여기선 (실패했지만) 하네스가 하려던 일.

즉 **하네스의 값 = 컨텍스트 전달.** 한계효용 ≈ P(정보 필요) × **P(달리 회수 안 됨)** × P(하네스가 가짐). 성숙한 repo의 정적 설계규칙에선 가운데 항 ≈ 0.

## ⚠️ 한계 / 방법론 결함 (정직)

- **결함(내 탓):** "비추론"을 *일반 지식* 기준으로만 골랐다 — *repo 기준* 비추론(코드에 없음)이어야 했다. 성숙한 repo라 거의 모든 계약이 이미 구현돼 있어, 정적 회수로 풀린다. 더 공정한 테스트는 *코드에 아직 없는 신규 임의 결정*인데, 그건 "내가 Arm A에만 알려준 값을 따르나"가 돼 *정보 컨텍스트 유무* 테스트로 변질(하네스가 아니라 컨텍스트를 잼). → 어느 쪽이든 결론은 같다: **하네스는 정보를 컨텍스트에 넣는 도구. 이미 회수되는 정보엔 0.**
- **n=1, 둘 다 천장.** 검정력 부족(동등성 증명 아님).
- **미측정 가치축:** (a) *효율/비용* — 짚어주면 *더 빨리/적은 토큰*으로 도달할 수 있음(정확도 아님). (b) *약한/드리프트 에이전트* — 덜 유능한 모델이나 긴 세션에선 회수 실패율↑ → 그때 주입이 살아날 수 있음. 둘 다 안 쟀다.

## 한 줄

> 의도적으로 *추론 불가*한 임의 Kafka 계약을 골랐는데도 **둘 다 5/5(null)** — 그 값들이 *기존 코드에 정적으로 존재*해 Arm B가 스스로 회수했기 때문. 4실험이 한 점으로 수렴: **하네스의 측정된 값은 *필요 정보가 정적 소스에 없을 때(런타임/환경/신규)*에만 양수, 정적으로 존재하면 유능한 모델이 회수해 redundant.** 정직한 함의 — 정적 설계문서 주입을 핵심으로 한 우리 하네스의 *정확도* 한계효용은 성숙한 코드베이스에서 ≈0이고, 방어 가능한 값은 (i) 런타임 grounding(v-next), (ii) 결정론 게이트, (iii) 미측정의 효율·약한에이전트 축으로 좁혀진다.
