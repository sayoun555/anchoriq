## 데이터 소스 확정

### API 실사용 가능성 검증

> AI가 11개 API 전부 한국 접근 가능 여부, 무료 티어 제한, Rate Limit 조사
> 일부 대체 필요한 것 발견:
> - ExchangeRate-API (월 100건 제한) → Frankfurter.app (완전 무료)
> - NOAA (미국 커버리지만) → Open-Meteo (글로벌)
> - ACLED (가입 복잡) → GDELT (무료, 키 불필요)

### 목업 데이터 논의

> AI가 화물, 보험, 소유구조 데이터는 목업으로 제안
> 사용자: "이거 필요한 데이터야?"
> AI: "솔직히 빼도 됩니다. 핵심이 아니에요."
> → 목업 없이 실제 API 데이터만으로 구성
> → "전부 실제 데이터입니다"가 더 강력한 포폴 어필

### 크롤링

> 사용자: "항만 혼잡도 크롤링하면 되지 크롤링 나 잘하잖아"
> → UNCTAD Port Tracker 크롤링으로 복원

## DB 설계 과정

### 그래프 DB 선택

> 사용자: "온톨로지 그래프 저게 좋아? 딥서치해서 최신이면서 가장 좋은게 뭔지 알아와봐"
> → AI가 Neo4j, Memgraph, Apache AGE, TypeDB 등 10개+ 비교 조사
> → Neo4j 선택 (Spring Boot 연동 최고, Cypher 쉬움, 팔란티어와 같은 모델)

### 4개 DB 확정

> PostgreSQL, Neo4j, Elasticsearch, Redis 4개 운용 결정
> Elasticsearch는 "빼도 되지만 넣으면 이력서에 한 줄 추가" → 가볍게 쓰기로

### 최적화 논의 — 하나씩 깊이 들어감

**Neo4j:**
> 사용자: "인덱스 많으면 좋은거 아닌가? 왜 복잡해져?"
> → 인덱스의 비용(쓰기 성능 저하) 설명
> → 선박 실시간 위치는 Neo4j 인덱스 안 걸고 Redis GEO로 분리하는 결정

**데이터 규모:**
> 사용자: "한국 입장에서 가장 위험한 항로로 해야지"
> → AI가 한국 해운 리스크 조사 (호르무즈, 말라카, 수에즈, 바브엘만데브, 대만해협, 파나마)
> → "아시아 발착 전체" 수만 척 + 4홉 깊이로 확정

**4홉 충분성:**
> 사용자: "4홉으로도 AI 의사결정 가능?"
> → 4홉으로 가능한 시나리오 전부 매핑해서 검증

**과최적화 경고:**
> AI: "솔직히 지금 단계에서 충분합니다. 실제로 느리지도 않은 걸 최적화하면 면접에서 마이너스."
> → Neo4j 최적화는 4가지로 제한, 진짜 병목(AIS, Kafka, AI 호출)에 집중

## AI 에이전트 활용 강점

### 1. 실시간 조사 + 즉시 의사결정

```
기존 방식: API 있나? → 구글링 2시간 → 정리 → 판단
AI 방식:   API 있나? → 11개 조사 + 한국 접근 가능 확인 → 5분 → 즉시 결정
```

### 2. 전문 지식 격차 해소

```
사용자: "온톨로지 개념을 잘 몰라서"
사용자: "Neo4j에 대해서는 하나도 모르거든?"
사용자: "파티셔닝에 대해서 잘 몰라서"
→ AI가 기존 지식(PostgreSQL)과 비교해서 즉석 설명
→ 이해 후 바로 의사결정
```

### 3. 빠짐 없는 검토

```
API 설계: 44개 → 86개 → 126개 → 174개 (6회 반복 검토)
"확실히 끝인가?" → 전체 기능 대조 → 빠진 것 발견
사람 혼자면 2~3회 검토 후 넘어갔을 것
```

### 4. 면접관 관점 사전 검증

```
AI가 시니어 면접관 역할:
- "혼자서 이걸 다 했다고요?" → 커밋 이력 깔끔하게
- "AI 바이브코딩 아니에요?" → 코드 설명 가능해야
- "Kafka 왜 썼어요?" → 정당한 이유 준비
→ 약점 사전 파악 + 보완 계획 (테스트, 부하 테스트, 모니터링, 문서화)
```

### 5. 일관성 유지

```
16개 문서 간 정합성:
- API 추가 → DB 스키마 업데이트 → IMPLEMENTATION_PLAN 반영
- 기술 결정 변경(JSoup → Playwright) → 관련 문서 전부 업데이트
사람 혼자면 문서 간 불일치 발생하기 쉬움
```

### 6. 의사결정 속도

```
2일 대화로 완성한 것:
- 프로젝트 기획
- 아키텍처 설계
- DB 스키마
- API 174개 설계
- Kafka 이벤트 설계
- Docker 인프라 설계
- 트랜잭션 전략
- 패키지 구조
- 코딩 규칙
- 구현 계획 11 Phase

사람 혼자 하면 최소 2~3주 소요 예상
```

### 7. job-crawler 패턴 재활용

```
AI가 기존 프로젝트(job-crawler) 문서 패턴 분석:
- SCREAMING_SNAKE_CASE 파일명
- 문제→원인→해결→CS원리→교훈 구조
- 한국어 + 영어 혼용 스타일
→ 동일 패턴으로 AnchorIQ 문서 작성 → 포트폴리오 일관성
```

## Day 3 — 아키텍처 패턴 확정 + 실무 트렌드 조사 (2026-03-28)

### WebFlux 논의

> 사용자: "웹플럭스 써야 하는거 아냐?"
> → 전면 WebFlux vs MVC + WebClient 비교
> → 사용자: "전부다 하는건 바보 같은거지 비동기들만 웹플럭스 하면 되는거 아냐?"
> → MVC + WebClient 하이브리드 확정

### R2DBC 논의

> 사용자: "그럼 R2DBC도 써?"
> → Virtual Threads가 있으니 R2DBC 불필요 설명
> → 사용자: "실무에서는 어떻게 하는지 딥서치해서 알아와"

### 실무 트렌드 대규모 조사

AI가 2025-2026 실무 트렌드 조사 수행:
- Netflix, 배달의민족, Toss, LG U+ 등 실제 기업 사례
- MVC vs WebFlux: Netflix도 메인에서 WebFlux 안 씀
- R2DBC: 실무에서 거의 안 씀 (JPA 못 쓰는 문제)
- Virtual Threads: 프로덕션 도입 확산 중 (Netflix 사례)
- 모듈러 모놀리스: 42% 기업이 MSA에서 회귀 (아마존 90% 비용 절감)
- CQRS: 자연스러운 분리가 대세 (Axon 같은 프레임워크 안 씀)
- 헥사고날: 원칙만 적용, 폴더 구조까지 바꾸지는 않음

> 사용자: "설계 다 갈아 엎어도 좋으니 어떤걸 쓰는게 좋은가? 가 중요해"
> → 조사 결과: 우리 설계가 이미 실무 트렌드와 일치
> → Virtual Threads 한 줄 추가만 필요

### 헥사고날 아키텍처 확정

> 사용자: "헥사고날로 해"
> → 폴더 이름 변경 vs 문서 명시 선택지 제시
> → 사용자: "문서로 명시"
